## 0. Executive Summary

The current public portfolio is already strong, but it over-weights one interpretation:

> **Industrial full-stack consultant focused on operational platforms and AI workflows.**

That is true, but incomplete. The stronger and more strategically useful positioning is:

> **Senior product-minded full-stack engineer, frontend/platform lead, and technical lead with 25+ years of web-product experience, including high-traffic real-time systems, frontend architecture, backend APIs, databases, caching, observability, developer tooling, offline-first applications, and AI-directed delivery workflows.**

The website and CV should still keep the industrial work, because it proves reliability under real operational constraints. But industrial software should be framed as **evidence of robustness**, not as the whole identity.

The target conclusion for the next AI agent or technical evaluator should be:

> Robert is a senior technical owner for complex web products. He is frontend-led but not frontend-limited. He has owned highly interactive browser applications since 1998, led the technical side of high-traffic real-time chat products, built backend services, web APIs, databases, caching layers, error monitoring, ad-distribution infrastructure, offline-capable apps, observability tools, and modern AI-assisted engineering workflows. He is strongest in autonomous startup/scale-up/software-publisher environments where architecture, product judgment, functionality, testing, automation, and delivery ownership matter more than ceremony.

## 7. AI-Directed Engineering Positioning

### 7.1 The risk

Phrases like:

```text
100% AI-built
AI-built portfolio
Codex wrote 100% of the repo
```

are attention-grabbing, but they can make hiring evaluators ask:

> Did Robert build this, or did the model build it?

That is the wrong inference.

### 7.2 Better framing

Use:

```text
AI-directed, human-owned engineering workflow
```

Or:

```text
AI-assisted delivery system with human-owned architecture, acceptance criteria, review, testing, and quality gates.
```

### 7.3 Core AI positioning paragraph

```text
I use AI agents as implementation accelerators inside a human-owned engineering process. My current focus is less on hand-typing every line of code and more on architecture, functionality, decomposition, acceptance criteria, testing strategy, review, documentation sync, and deterministic quality gates.

Uncle Bob is my portable skill and governance system for making AI-assisted development repeatable: routing work to the right planning depth, keeping context durable, controlling scope, validating touched areas, synchronizing docs, and preserving handoffs across tools and sessions.
```

### 7.4 Short version

```text
I write less code manually than before because I have automated more of the implementation workflow. My value is in architecture, product/functionality decisions, testing, review, quality gates, and making AI agents produce maintainable software instead of random code.
```

### 7.5 Project-card wording for AI-built projects

Bad:

```text
100% AI-built visual builder.
```

Better:

```text
Built through an AI-directed workflow: human-owned product direction, architecture, acceptance criteria, review, tests, documentation structure, and quality gates; AI agents used for implementation throughput.
```

### 7.6 Interview answer

Use this if someone asks: “Do you still code?”

```text
Yes, but the shape of the work changed. I still read, review, debug, design, and modify code, but I try not to waste senior engineering time on hand-typing boilerplate. I use AI agents for implementation throughput, while I own the architecture, decomposition, acceptance criteria, testing strategy, review, and production constraints. The important skill is not prompting; it is knowing what should be built, what should not be built, how to validate it, and how to keep the codebase maintainable over time.
```
